# Rollback and Recovery

## Goal
Дать единый порядок отката релиза и восстановления после сбоев для backend MVP (`apps/api` + PostgreSQL).

## When to Use
- После неуспешного релиза с критичным влиянием на заказы/доступность.
- При нарушении инвариантов данных (`orders`, `inventory`).
- При необходимости восстановить БД из backup.

## Application Rollback
1. Остановить rollout и зафиксировать инцидент.
2. Определить последнюю стабильную версию.
3. Выполнить rollback на предыдущий стабильный артефакт.
4. Проверить:
   - `GET /health/db`
   - `GET /products`
   - `POST /orders` (smoke)
5. Сообщить статус по процессу из `docs/runbooks/incident-response.md`.

## Database Recovery (High-level)
1. Подтвердить момент сбоя и выбрать recovery point.
2. Восстановить БД из проверенного backup согласно инфраструктурному регламенту.
3. Проверить доступность БД и Prisma-схемы (`db:validate`).
4. Выполнить checks консистентности заказов/остатков.

## Consistency Checks
- Нет заказов со статусом `SHIPPED`, где не списан `onHand`.
- Нет отрицательных `reserved` или `onHand`.
- Переходы в `OrderStatusHistory` соответствуют допустимой матрице статусов.
- Ошибки API возвращают ожидаемые контрактные коды.

## Exit Criteria
- Сервис стабилен в окне наблюдения.
- Критичные проверки прошли без регрессий.
- Сформирован postmortem с action items и сроками.
