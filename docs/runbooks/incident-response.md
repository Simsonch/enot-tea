# Реагирование на инциденты (MVP)

## Цель

Задать единый регламент реагирования на production-инциденты в MVP: быстро стабилизировать сервис, минимизировать влияние на пользователей и обеспечить постинцидентные улучшения.

## Область

- Применяется к backend MVP в `apps/api` и связанным сценариям `orders`/`inventory`.
- Используется для инцидентов после релиза и в штатной эксплуатации.
- Для rollback/recovery шагов использовать `docs/runbooks/rollback-and-recovery.md`.
- Для release-контекста и go/no-go использовать `docs/runbooks/release-process.md`.

## Уровни критичности

- `SEV-1`: недоступен checkout/заказы, есть риск потери данных.
- `SEV-2`: сильная деградация, но есть обходной путь.
- `SEV-3`: некритичная ошибка с низким бизнес-влиянием.

## Роли и ответственности

- Incident Commander (IC):
  - владеет инцидентом от начала до завершения;
  - принимает решения rollback/forward-fix;
  - подтверждает уровень критичности и закрытие инцидента.
- Ops Driver:
  - выполняет технические шаги (rollback, hotfix, проверки);
  - ведет таймлайн событий и фиксирует фактические команды/действия;
  - подтверждает техническую стабилизацию.
- Communications Owner:
  - публикует статус-апдейты по расписанию;
  - синхронизирует ожидания по ETA и next steps;
  - фиксирует финальный статус и ссылку на postmortem.
- Domain Owner (`orders`/`inventory`):
  - оценивает бизнес-влияние и корректность контрактов;
  - подтверждает восстановление критичных пользовательских сценариев.
- Tech Lead (Escalation Owner):
  - подключается при `SEV-1` сразу и при затяжных `SEV-2`;
  - снимает блокеры по ресурсам и приоритизации.

## Incident Flow

1. **Detect**: подтвердить сигнал (alert/health-check/бизнес-метрика) и открыть инцидент.
2. **Assess**: определить `SEV` и затронутые функции.
3. **Assign**: назначить IC, Ops Driver, Communications Owner, Domain Owner.
4. **Mitigate**: ограничить влияние (rollback/feature toggle/hotfix/read-only режим).
5. **Stabilize**: подтвердить восстановление ключевых сценариев.
6. **Close**: объявить стабилизацию и открыть postmortem.

## Шаги реагирования (операционный чеклист)

1. Подтвердить масштаб и критичность.
2. Назначить роли IC/Ops/Communications/Domain Owner.
3. Зафиксировать: время начала, симптом, первичную гипотезу причины.
4. Определить стратегию: rollback или forward-fix.
5. Выполнить mitigation и перепроверить:
   - `GET /health/db`;
   - критичный сценарий `orders` (`POST /orders`, `PATCH /orders/:id/status`);
   - базовую доступность `GET /products`.
6. Подтвердить стабилизацию по метрикам и логам.
7. Зафиксировать root cause (или текущую рабочую версию причины), impact и action items.

## Эскалация

### Matrix
- `SEV-1`:
  - немедленная эскалация: IC + Tech Lead + Product Owner;
  - старт mitigation без ожидания полной диагностики;
  - решение rollback/forward-fix в первые 15 минут.
- `SEV-2`:
  - эскалация Tech Lead при отсутствии стабилизации в течение 60 минут;
  - при ухудшении влияния перевести в `SEV-1`.
- `SEV-3`:
  - обработка в рабочем порядке;
  - обязательный баг-тикет с owner и сроком;
  - при росте влияния перевести в `SEV-2`.

### Escalation Triggers
- Увеличение затронутой функциональности или аудитории.
- Рост `5xx`/contract error rate в `orders`.
- Риск нарушения инвариантов данных (`onHand`, `reserved`, `statusHistory`).

## Коммуникации

- Для `SEV-1`: апдейты каждые 30 минут до стабилизации.
- Для `SEV-2`: апдейты каждый час до стабилизации.
- Минимум в апдейте:
  - текущий статус и затронутый функционал;
  - предпринятые действия;
  - следующий шаг и ETA.

## Decision Policy: Rollback vs Forward-fix

- Выбирать **rollback**, если:
  - недоступны критичные API или деградирован основной order flow;
  - есть риск потери/порчи данных;
  - безопасный фикс не укладывается в короткое окно.
- Выбирать **forward-fix**, если:
  - влияние ограничено и контролируемо;
  - есть проверенный фикс с низким риском;
  - IC и Tech Lead согласовали стратегию и план верификации.

## Минимальные источники алертов

- Проверки доступности health endpoint API.
- Логи ошибок приложения.
- Алерты на падение доли успешного создания заказов.

## Exit Criteria for Incident

- Сервис стабилен в согласованном окне наблюдения.
- Критичные проверки проходят без деградации.
- Статус инцидента официально закрыт IC.
- Создан postmortem с action items.

## Postmortem Policy

- Для `SEV-1`: postmortem обязателен в течение 24 часов.
- Для `SEV-2`: postmortem обязателен в течение 72 часов.
- Для `SEV-3`: RCA по решению Tech Lead (минимум баг-тикет обязателен).

## Postmortem Template (minimum)

- Incident ID и период.
- Краткое влияние на пользователей/бизнес.
- Таймлайн ключевых событий.
- Root cause и contributing factors.
- Что сработало хорошо.
- Что нужно улучшить.
- Action items: owner + deadline.
- Предотвращение повторения: какие контрольные проверки добавлены (тесты, алерты, runbook).
