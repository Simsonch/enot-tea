# Обзор проекта

## Цель
Enot Tea - это платформа электронной коммерции для продажи чайной продукции с публичной витриной и внутренней админ-панелью.
Система закрывает ежедневные операционные задачи для клиентов, консультантов, менеджеров и владельца: каталог, заказы, склад и чат поддержки в одном месте.
Цель MVP - запустить надежную первую версию со сквозной обработкой реальных заказов и возможностью дальнейшего расширения AI-сценариями поддержки.

## Границы
- Входит в MVP:
  - Публичная витрина с каталогом, карточками товаров и оформлением заказа.
  - Админ-панель для обработки заказов, контроля остатков и управления товарами.
  - Backend API с аутентификацией, ролевым доступом и ключевыми бизнес-модулями.
  - Схема PostgreSQL и миграции для пользователей, товаров, склада, заказов и чата.
  - Чат клиента с консультантом в базовом MVP-виде.
- Не входит в MVP:
  - Нативные мобильные приложения.
  - Продвинутая аналитика и BI-дашборды.
  - Сложная оркестрация платежных шлюзов и бухгалтерские интеграции.
  - Мультисклад и мультистрановые юридические/налоговые сценарии.
  - Полностью автономный AI-агент с выполнением действий.

## Ключевые функции
- Каталог витрины с поиском, карточкой товара и созданием заказа.
- Админский конвейер заказов с переходами статусов и историей.
- Учет складских остатков в разрезе `onHand` и `reserved`.
- Ролевой доступ для владельца, менеджера, консультанта и клиента.
- Базовый чат в реальном времени между клиентом и консультантом.

## Базовая техническая архитектура (MVP)
- Монорепозиторий с разделением на `apps/*` (деплоимые приложения) и `packages/*` (общие контракты и библиотеки).
- Управление workspace и зависимостями выполняется через `pnpm` (`pnpm-workspace.yaml`).
- Стартовые приложения: `apps/api`, `apps/storefront`, `apps/admin`; общий модуль: `packages/shared`.
- Backend в `apps/api` использует NestJS + Node.js + TypeScript с Prisma Migrate и Prisma Client для работы с PostgreSQL.
- Базовый runtime для локальной разработки: Node `24.15.0` LTS (см. `.nvmrc`).
- Локальная PostgreSQL-инфраструктура стандартизирована через Docker Compose в корне репозитория.

## Термины домена
- Product: товарная позиция каталога с SKU, ценой и метаданными.
- Inventory Item: запись складского учета с количеством `onHand` и `reserved`.
- Order: клиентский заказ с позициями, суммами и жизненным циклом статусов.
- Order Status: этап обработки заказа (`NEW`, `CONFIRMED`, `PACKED`, `SHIPPED`, `DELIVERED`, `CANCELLED`).
- Reservation: временный резерв товара при обработке заказа.
- Consultant Session: диалог поддержки между клиентом и консультантом.

## Нефункциональные требования
- Производительность:
  - Типовые запросы чтения API должны отвечать до 300 мс при MVP-нагрузке.
  - Первая отрисовка витрины должна укладываться в 2.5 с на широкополосном соединении.
- Надежность:
  - Критические сценарии обработки заказа должны быть восстановимыми и аудируемыми.
  - Миграции должны воспроизводиться в чистом окружении.
  - Для локальной разработки инфраструктура PostgreSQL стандартизирована через Docker Compose.
- Безопасность:
  - Секреты не хранятся в репозитории; используются только переменные окружения.
  - Валидация входных данных на границах API и ролевой контроль доступа обязательны.
  - Чувствительные данные и токены не должны попадать в логи.

## Метрики успеха
- Доля завершенных checkout-сценариев из корзины >= 50% на начальном трафике.
- >= 95% заказов проходят через админку без ручного вмешательства в БД.
- Инциденты рассинхронизации остатков < 1% от обработанных заказов.
- Инциденты production-уровня из-за регрессий деплоя = 0 в период запуска MVP.
