# Реагирование на инциденты (MVP)

## Цель

Задать минимальный регламент реагирования на инциденты, влияющие на production в период MVP.

## Уровни критичности

- `SEV-1`: недоступен checkout/заказы, есть риск потери данных.
- `SEV-2`: сильная деградация, но есть обходной путь.
- `SEV-3`: некритичная ошибка с низким бизнес-влиянием.

## Роли

- Incident Commander (IC): ведет инцидент, принимает решения по rollback/forward-fix.
- Ops Driver: выполняет технические шаги и фиксирует таймлайн.
- Communications Owner: обновляет статус для команды и стейкхолдеров.
- Domain Owner: подтверждает влияние на бизнес-функции (`orders`, inventory, checkout).

## Шаги реагирования

1. Подтвердить масштаб и критичность.
2. Назначить IC, Ops Driver, Communications Owner.
3. Ограничить влияние (rollback/feature toggle/hotfix).
4. Для SEV-1 обновлять статус каждые 30 минут.
5. Определить стратегию: rollback или forward-fix.
6. Зафиксировать root cause и список action items.

## Эскалация

- `SEV-1`: немедленная эскалация IC + Tech Lead + Product Owner.
- `SEV-2`: эскалация Tech Lead при отсутствии стабилизации в течение 60 минут.
- `SEV-3`: обработка в рабочем порядке, обязательный баг-тикет и приоритизация.

## Коммуникации

- Для `SEV-1`: апдейты каждые 30 минут до стабилизации.
- Для `SEV-2`: апдейты каждый час до стабилизации.
- Минимум в апдейте:
  - текущий статус и затронутый функционал;
  - предпринятые действия;
  - следующий шаг и ETA.

## Минимальные источники алертов

- Проверки доступности health endpoint API.
- Логи ошибок приложения.
- Алерты на падение доли успешного создания заказов.

## Postmortem Template (minimum)

- Incident ID и период.
- Краткое влияние на пользователей/бизнес.
- Таймлайн ключевых событий.
- Root cause и contributing factors.
- Что сработало хорошо.
- Что нужно улучшить.
- Action items: owner + deadline.
